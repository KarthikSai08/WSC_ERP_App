What I Built — The WSC ERP System

I built a multi-module Enterprise Resource Planning (ERP) system from scratch using .NET 10, C# 14, and Clean Architecture. The system is split into 6 microservice modules — Store, CRM, Delivery, Dashboard, Shared, and Gateway — each following a strict 4-layer structure: Domain → Application → Infrastructure → API. It uses Dapper for raw SQL data access, Redis for caching, and SQL Server as the primary database. The modules communicate via shared contracts and typed HTTP clients.

Design & Architectural Patterns Applied

Architecture

🏛 Clean Architecture (Onion)

Domain → Application → Infrastructure → API layers with inward-only dependency flow. Each module is a self-contained vertical slice.

Design Pattern

🏭 Repository Pattern

IGenericRepository<T> abstraction with concrete Dapper implementations. Entity-specific repositories extend the generic base.

Design Pattern

🔄 Unit of Work Pattern

Coordinates multiple repositories under a single transaction. IUnitOfWork exposes all repos and manages commit/rollback consistently.

Data Pattern

⚡ Cache-Aside Pattern

Check Redis first → on miss, hit DB and write back to cache. Used in Store and Delivery modules via RedisCacheService.

Architecture

🔌 Dependency Injection (IoC)

All dependencies registered via IServiceCollection extensions. Program.cs is the composition root, no service locator used.

API Pattern

🌐 RESTful API Design

Resource-based URLs, proper HTTP verbs, action result types (200, 201, 400, 404). Exception middleware for consistent error format.

Design Pattern

🗺️ DTO Mapping Pattern

Entities never leak to API consumers. AutoMapper Profile classes handle Entity ↔ DTO transformation in the Application layer.

Architecture

📦 Shared Kernel Pattern

WSC.Shared.Contracts contains cross-module DTOs, exceptions, interfaces, and enums. WSC.Shared.Infrastructure provides typed HTTP clients.

Resilience

🛡️ Global Exception Middleware

Single try/catch at the request pipeline boundary. Catches domain and infra exceptions and converts them to structured ProblemDetails responses.

Data Pattern

📋 Stored Procedure Pattern

Complex operations (multi-table inserts, atomic updates) encapsulated in stored procs. Dapper calls them with DynamicParameters.

API Pattern

🔢 API Versioning (Planned)

Controller folders structured as v1/ and v2/. Route versioning middleware to be activated when features stabilize.

Design Pattern

🔑 Options Pattern

appsettings.json values bound to typed options classes (connection strings, Redis config, JWT settings) via IOptions<T>.

Key Lessons & Takeaways

🧱 Structure First, Code Second

Getting the Clean Architecture layers right before writing business logic saved massive refactoring effort later. Investing in folder structure pays off.

🔗 Abstractions Enable Testability

Every repository and service behind an interface means the code can be unit-tested with mocks. Skipping interfaces makes code rigid and hard to test.

🗃 Dapper > EF Core for Control

Dapper gives full SQL control and is perfect for stored-procedure-heavy enterprise apps. EF Core is better for rapid CRUD with migrations — know when to use which.

📦 Shared Contracts Prevent Coupling

Putting DTOs and interfaces in WSC.Shared.Contracts lets modules communicate without knowing each other's internals. This is how real microservices scale.

⚡ Cache-Aside is Simple but Powerful

The pattern itself is easy: check Redis, miss → DB, write back. The hard part is cache invalidation and key design — patterns I need to practice more.

🛡 Exception Middleware Matters

Having a single global exception handler means API consumers always get consistent error shapes. Scattering try/catch in controllers leads to inconsistent errors.

📋 Documentation While Building

Creating PROJECT_STRUCTURE.html as I built helped cement my understanding of what I was constructing and made architecture decisions more deliberate.

🔄 Incremental Complexity Works

Starting with Domain → Infra → App → API before adding Redis, JWT, and Rate Limiting meant every new layer had a stable foundation to build on.
