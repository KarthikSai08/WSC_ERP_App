📍 Tech Stack Implementation Guide

A quick reference for where each technology should be implemented:

Layer-by-Layer Implementation

🌐 API Layer (Controllers & Middleware)

  • JWT Authentication: AuthenticationMiddleware, JWT tokens generation
  • Global Exception Middleware: ErrorHandlingMiddleware for centralized error handling
  • Rate Limiting: RateLimitingMiddleware to prevent abuse
  • API Versioning: Route versioning (v1, v2) in controllers
  • Auditing: Audit logging endpoints and user actions

⚙️ Application Layer (Services & Business Logic)

  • Caching Logic (Cache-Aside Pattern): Check cache → if miss → fetch from DB → store in cache
  • Idempotency: IdempotencyService to track & prevent duplicate requests
  • Auditing: Log business operations and changes
  • Use Cases: Business logic coordination

💾 Infrastructure Layer (Data Access & Persistence)

  • Dapper ORM: DapperContext, data mapping, and query execution
  • Dynamic Parameters: DynamicParameters for parameterized queries
  • Stored Procedures: Call SPs via Dapper for complex queries
  • Transactions: TransactionManager for ACID operations
  • Unit of Work Pattern: UnitOfWork class managing repositories & transactions
  • Redis Caching: CacheRepository for cache operations
  • Repositories: Generic Repository pattern with Dapper
  • Schema Separation: Different schemas (dbo, audit, cache) in database

🔗 Shared & Infrastructure (Cross-Cutting Concerns)

  • Serilog Logging: Structured logging across all layers
  • JWT Configuration: TokenProvider service for token generation
  • Auditing Service: AuditService to log changes and user actions
  • Exception Handling: Custom exception classes & handlers
  • Constants & Configurations: Shared settings for all modules

🏗️ Architecture Overview

The application follows Clean Architecture with a Microservices approach. Each module contains layered structure:

Layered Architecture Pattern

1️⃣ API Layer (Entry Point)

  • REST API endpoints
  • Request/Response handling
  • Program.cs configuration
  • Dependency injection setup

2️⃣ Application Layer (Business Logic)

  • Use cases and workflows
  • Application services
  • DTOs (Data Transfer Objects)
  • Business logic orchestration

3️⃣ Domain Layer (Core Business)

  • Entities (business objects)
  • Value Objects
  • Enums (constants)
  • Business rules
  • Domain services

4️⃣ Infrastructure Layer (Technical Details)

  • Database access (EF Core)
  • External service integrations
  • Repository implementations
  • Configuration

📋 Development Guidelines

🏛️ Architecture Principles

  • Clean Architecture: Keep concerns separated in distinct layers
  • DDD (Domain-Driven Design): Focus on business logic in domain layer
  • SOLID Principles: Follow single responsibility, dependency inversion, etc.
  • Microservices: Each module can run independently
  • Service-Oriented: Business logic centralized in application services

📁 File Organization

  • Domain/ - Business entities, value objects, enums, business rules
  • Application/ - Use cases, services, DTOs, application logic
  • Infrastructure/ - Data persistence, external services, configuration
  • API/ - Controllers, middleware, entry point (Program.cs)

🔄 Communication Flow

  • Request: API Controller → Application Service → Domain Logic → Infrastructure Repository
  • Response: Repository → Domain Logic → Application Service → API Controller
  • Cross-module: Use Shared.Contracts for common models
